# Audit Keamanan Verdix — 2026-07-21

Metode: Slither 0.11.5 (statik, 9 kontrak `src/`) + 3 agent review manual paralel
(escrow/payment, vault/factory, registry/token/oracle/memory), tiap temuan
diverifikasi adversarial dengan skenario eksploit konkret. READ-ONLY.

## Verdict dua thesis inti

| Thesis | Status | Catatan |
|---|---|---|
| **#1 Policy vault di-enforce on-chain (agent tak bisa override)** | ✅ **LOLOS** | Agent non-owner tak bisa nembus policy / nyolong dana. Slither "arbitrary-send" = false-positive (target dibatasi allowlist owner-only). |
| **#2 Economic Memory tak bisa dipalsukan** | ⚠️ **BOLONG** | Pihak **eksternal** tak bisa palsu (record() di-gate recorder, setAgentWallet butuh signature). Tapi **operator sendiri** bisa fabrikasi track record (HIGH-1), dan integritas ledger nyandar 1 owner EOA (HIGH-2). |

Ringkas: **dana aman, reputasi belum sepenuhnya.** Yang jual Verdix (track
record jujur) punya lubang buat aktor yang memalsukan reputasi *dirinya sendiri*
— wajib kefix sebelum onboarding operator eksternal / mainnet.

## HIGH

### HIGH-1 — Wash-trading: fabrikasi Economic Memory lewat self-dealing + act() value-0
`src/RiskGuardVault.sol:108-137`, `src/VaultFactory.sol:167-188`
Tiap `act()` yang lolos mencatat entry Tier-1 Settlement `Outcome.Success`, tapi
vault tak pernah memverifikasi transfer ke pihak independen. Tiga celah gabung:
(1) `target` cukup ada di `allowedTarget` — bisa EOA milik attacker; (2) `value=0`
lolos semua cek (`0>maxTx` false, `spent+0<=cap`, `balance-0>=floor`, `call{value:0}`
return success); (3) `counterpartyId` hardcoded `0`, `outcome` hardcoded `Success`.
Karena `act` revert saat transfer gagal, EconomicMemory **secara konstruksi tak
pernah** mencatat kegagalan → success-rate dari vault selalu 100%.
**Eksploit:** Mallory register agent → createVault → setTarget(malloryEOA,true) →
spam `act(malloryEOA, 0, memo)` (cooldown=0). Reputasi "100% sukses" dengan biaya
gas doang, nol aktivitas ekonomi riil.
**Fix:** wajibkan `value >= minSettlement`; rekam `counterpartyId` agent lain &
larang `target` yang controller-nya sama dengan controller managerAgent.

### HIGH-2 — Integritas ledger reputasi bertumpu 1 owner EOA (sentralisasi)
`src/EconomicMemory.sol:79-104`
`record()` tak memverifikasi hubungan recorder↔agent: recorder ter-whitelist bisa
menulis entry apa pun untuk agent mana pun; owner bisa `setRecorder` alamat apa
pun termasuk EOA-nya. Owner key bocor/jahat → seluruh klaim "reputasi tak bisa
dipalsukan" runtuh. (Non-owner tetap aman — terbukti test.)
**Fix:** pindah owner EconomicMemory & StressOracle ke multisig + timelock;
dokumentasikan keamanan sebenarnya = proses review recorder, bukan kode.

## MEDIUM

- **M1 Front-running expire vs dispute (curi worker-bond)** — `TaskEscrow.sol:198,127`.
  `confirm` tak cek deadline; `expire` terbuka utk siapa saja & selalu memihak
  client (refund payment+clientBond+**workerBond**). Client diam sampai deadline,
  front-run `dispute()` worker dengan `expire()` gas lebih tinggi → kerja gratis +
  rampas bond. Fix: task Accepted lewat-deadline paksa ke jalur Disputed / kasih
  worker grace-window.
- **M2 Dispute tanpa timeout → dana terkunci permanen** — `TaskEscrow.sol:170,79,72`.
  Status Disputed cuma keluar via `rule()` onlyArbitrator; tak ada fallback.
  Arbitrator di-set `address(0)` (tanpa zero-check) / kunci hilang → payment+2 bond
  nyangkut selamanya. Fix: zero-check + timeout fallback refund.
- **M3 De-auth recorder membekukan escrow in-flight** — `TaskEscrow.sol:136,184,209`.
  confirm/rule/expire manggil `record()` yang revert kalau escrow bukan recorder.
  `setRecorder(escrow,false)` saat ada task Accepted → 3 jalur exit revert → dana
  stuck. Fix: bungkus `record` try/catch, atau larang cabut recorder saat ada task
  terbuka.
- **M4 Stake pihak ketiga dirampas controller baru pasca-transfer NFT** —
  `VDXStaking.sol:63-83` + `AgentRegistry.sol:169-175`. Voucher stake utk agent A;
  A jual NFT ke M; `stakedOf` tetap → M jadi controller → M sapu stake voucher.
  Fix: catat staker per (agentId,address), staker cuma bisa tarik miliknya.
- **M5 act() tanpa reentrancy guard** — `RiskGuardVault.sol:124`, `VaultFactory.sol:183`.
  CEI sebagian benar (spent/lastAction di-update sebelum call), drain tercegah, tapi
  target-kontrak + cooldown=0 bisa re-enter → burst dalam 1 tx lewati maksud
  cooldown. Fix: `nonReentrant` di act() & withdraw().
- **M6 recordAction percaya arg agentId, bukan menurunkannya** — `VaultFactory.sol:54-60`.
  Aman sekarang (cuma GuardedVault.act pemanggil), tapi begitu ada varian vault lain
  di-isVault, ia bisa nulis reputasi atas agentId korban. Fix: derive agentId dari
  `GuardedVault(msg.sender).managerAgentId()`.

## LOW (ringkas)
- setAgentWallet tanpa nonce → replay dalam deadline (`AgentRegistry.sol:112`).
- `WalletInUse` dideklarasi tapi tak di-enforce → 1 wallet kontrol banyak agent (sybil).
- owner vault tak immutable & tak ada transferOwnership → kunci hilang = dana terkunci.
- EconomicMemory.setOwner tanpa zero-check (setOwner(0) = brick) & tanpa event.
- createVault tanpa cek controller → grief atribusi listing (kosmetik).
- owner withdraw() nembus haltFloor (by-design; jangan klaim floor proteksi mutlak).
- cooldown pakai block.timestamp (manipulasi validator ~detik; minor).
- StressOracle owner immutable tanpa transfer path.

## Yang sehat (verified positif)
1. **Enforcement anti rogue-agent solid**: act() cek isController→allowedTarget→
   maxTx→dailyCap(reset window benar)→cooldown→haltFloor sebelum transfer; semua
   dites (NonWhitelistedTarget, MaxTx, DailyCap, Cooldown, HaltFloor, fuzz NeverBelowFloor).
2. **VDX mint aman**: fixed 500M di constructor, tak ada fungsi mint, cuma Burnable.
3. **Identitas tahan spoof**: setAgentWallet butuh signature wallet target, digest
   ikat chainId+contract+agentId (anti replay lintas-chain/kontrak/agent).
4. **Escrow konservasi ETH & CEI konsisten**: status→terminal sebelum _send (reentrancy
   lintas-fungsi revert BadStatus); testFuzz_EscrowConservesEther saldo=0 pasca-settle.
5. **Aritmetika aman**: solc 0.8 tanpa unchecked di jalur dana; cast uint128 di bawah
   batas yang dijamin; underflow floor revert.

## Prioritas sebelum onboarding operator eksternal / mainnet
1. **HIGH-1** wash-trading (nilai jual Verdix — tanpa ini track record gak kredibel)
2. **HIGH-2** owner EOA → multisig+timelock (governance core asset)
3. **M1 + M2** escrow: front-running bond + dispute timeout (uang operator nyata)
4. **M5 + M6** reentrancy guard + derive agentId (defense-in-depth core asset)
5. Sisa MED/LOW saat cleanup pra-mainnet.
